# SOPS + age 비밀정보 관리 체계 도입 보고서

## 1. 보고 목적

본 문서는 현재 프로젝트에 적용한 SOPS + age 기반 비밀정보 관리 체계를 기준으로,

- 왜 이 체계를 채택했는지
- 사용자(개발자) 설치/사용 경험이 어떻게 바뀌는지
- 조직 전체 확산 시 어떤 방식으로 추진할지

를 상급자 의사결정 관점에서 정리한 보고서입니다.

---

## 2. 현재 적용 완료 범위

이번 작업으로 아래 항목을 실제 레포에 반영했습니다.

- `.sops.yaml` 기반 암호화 정책 구성
- `.gitignore`에서 평문 `.env*` 차단, `.env*.enc` 추적 허용
- `pre-commit` 훅으로 평문 `.env*` staged 시 커밋 차단
- IDE 실행 시 자동 환경변수 주입 스크립트 (`scripts/open-ide.sh`)
- 일반 명령 실행 시 자동 환경변수 주입 스크립트 (`scripts/run-with-env.sh`)
- 운영 문서 (`README.md`, `docs/secret-management.md`) 정비
- 설치/초기설정 자동화 스크립트 분리 (`install/`)
  - `install/setup-secrets-windows.bat`
  - `install/setup-secrets-macos.sh`
  - `install/setup-secrets-linux.sh`

## 3. 핵심 설계 방향

### 3.1 기본 원칙

1. Git에는 암호화 파일만 저장한다.
2. 평문 `.env`는 로컬 전용으로 관리한다.
3. 개인키는 사용자 로컬에만 존재한다.
4. 팀 권한 변경은 공개키(recipient) 갱신으로 처리한다.

### 3.2 팀원 권한 변경 표준

- 팀원 추가: `sops updatekeys`
- 팀원 제거: `sops rotate -i` (데이터 키 재생성 필수)

이 기준으로 권한 변경 이력과 보안 수준을 일관되게 유지합니다.

---

## 4. 자동 암복호화/주입 전략 (중점)

### 4.1 자동 “주입” 방식 채택 이유

이번 설계는 **평문 파일을 항상 만들어두는 방식**보다,
**실행 시점에만 복호화하여 프로세스 환경변수로 주입**하는 방식을 우선합니다.

- IDE 실행: `scripts/open-ide.sh`
- 앱/명령 실행: `scripts/run-with-env.sh`
- 내부 동작: `sops exec-env` 사용

### 4.2 기대 효과

- 평문 파일 잔존 위험 감소
- 실행 환경 일관성 증가 (개발자별 환경 오차 축소)
- “실수로 평문 커밋” 가능성 구조적 축소

### 4.3 자동 “암호화” 처리 원칙

- 기본 정책: 커밋 차단 모드(안전 우선)
- 선택 정책: `AUTO_ENCRYPT_ENV=1` 기반 자동 갱신(편의 보조)

권장안은 자동 암호화 의존보다, 암호문 갱신을 명시적으로 수행하고 리뷰하는 방식입니다.

## 5. 사용자 설치/온보딩 방식

### 5.1 기존 방식의 문제

- 도구 설치와 PATH 설정이 개발자별로 달라 초기 진입장벽이 높음
- `.env` 취급 방식이 팀원마다 달라 실수 확률 증가

### 5.2 개선된 방식

운영체제별 설치 스크립트 1회 실행으로 초기 구성이 자동화됩니다.

- Windows:
  - `.\install\setup-secrets-windows.bat`
- macOS:
  - `chmod +x ./install/setup-secrets-macos.sh && ./install/setup-secrets-macos.sh`
- Linux:
  - `chmod +x ./install/setup-secrets-linux.sh && ./install/setup-secrets-linux.sh`

자동 처리 항목:

1. `sops`, `age`, `age-keygen` 설치
2. PATH 등록
3. age 키 생성
4. `SOPS_AGE_KEY_FILE` 설정
5. 공개키 출력

### 5.3 사용자 관점 사용성 차이

- 도입 전: 문서 따라 수동 설치/설정, 오류 발생 시 개인 해결
- 도입 후: 설치 스크립트 1회 실행 + 공개키 공유 + 바로 개발 가능

결과적으로 온보딩 시간이 단축되고, 환경 편차가 줄어듭니다.

## 10. 결론

이번 도입안은 “보안 강화”와 “개발 편의성”을 동시에 만족하는 방향입니다.

- 보안: 평문 유입 차단 + 키 기반 권한 통제
- 사용성: 설치 자동화 + 실행 시 자동 주입
- 확장성: 템플릿/일괄 PR 방식으로 조직 전체 확산 가능

즉, 개별 수작업 중심 운영에서 벗어나 **표준화된 자동화 기반 운영 체계**로 전환할 수 있습니다.
